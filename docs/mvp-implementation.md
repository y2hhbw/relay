# Relay MVP 实现状态（清晰版）

本文档用于明确三件事：

1. 目前已经完成了什么
2. 目前还没完成但非常必要的是什么
3. 按优先级下一步该做什么

更新时间：2026-03-13

## 一、已完成（可运行能力）

### A. 账户与余额主链路（已完成）

- 已支持创建账户、生成 API key、生成储值地址
- 已支持余额查询（可用余额 / 冻结余额）
- 已支持入账幂等（`tx_hash + log_index`）
- 已支持账本写入（`ledger_entries`）

### B. 计费与调用主链路（已完成）

- 已支持固定价格调用（`search.web`、`ocr.parse_image`）
- 已支持 LLM 冻结后结算（reserve -> settle -> release）
- 已支持调用审计持久化（`api_calls`）
- 已支持调用查询：
  - 账户隔离
  - `service_key` / `status` 过滤
  - `start_at` / `end_at` 时间范围过滤
  - `limit` / `offset` 分页
  - cursor 分页（已升级为稳定键游标，基于 `api_calls.sequence_id`）

### C. 基础安全与稳定性（已完成）

- 已支持 API key 鉴权
- 已支持请求 schema 严格校验（缺失字段、未知字段返回 422）
- 已支持进程内时间窗口限流（按 account + service）

### D. 链上 listener（已完成 MVP 版）

- 已支持链上 USDC Transfer 事件轮询
- 已支持区块状态持久化（state file）
- 已复用现有入账逻辑（命中充值地址即入账）
- 已支持失败重试退避
- 已支持连续失败阈值告警
- 已支持告警冷却（防止告警刷屏）
- 已支持运行指标快照（成功/失败/连续失败/最近成功时间）
- 已支持 webhook 告警下沉（可选）

## 二、尚未完成但非常必要（按优先级）

### P0（必须尽快做，影响可外测）

1. 接入至少一个真实上游 provider（不仅是 mock）
   - 现状：`search.web` 仍允许 mock 模式
   - 风险：无法验证真实外部依赖下的稳定性和成本

2. 把限流从进程内切到 Redis
   - 现状：进程内限流重启即丢失，多实例不共享
   - 风险：横向扩展后限流失效或不一致

3. 链上 listener 增加生产级观测
   - 现状：有日志、告警、指标快照，但没有统一指标暴露
   - 风险：线上排障效率低

### P1（很重要，建议紧随 P0）

1. 对 cursor 增加签名/HMAC 防篡改
   - 现状：cursor 可解码但未签名
   - 风险：恶意构造参数导致异常查询模式

2. 为 listener 增加外部告警通道策略
   - 现状：支持单 webhook
   - 风险：单点告警通道不稳定

3. API 调用查询性能继续优化
   - 现状：已加核心索引
   - 风险：高数据量下仍需压测和索引微调

### P2（增强项，放在 P0/P1 后）

1. 更完整的运营接口（调用报表、聚合统计）
2. 更完整的 listener 运维能力（灰度、多链）
3. 更细粒度的计费配置与策略治理

## 三、当前对外能力边界（务实说明）

当前系统已经是“可运行 MVP”，但仍属于“可外测前夕”而非“生产完备”：

- 可以验证核心价值：储值 + API 调用 + 计费 + 审计 + 链上入账
- 还不建议直接高流量生产：主要缺口在多实例限流、生产级观测与真实 provider 完整性

当前阶段结算策略（已确认）：

- 用户消费金额先统一结算到平台账内
- 平台与服务提供方采用线下/手工对账结算
- 自动化 provider 出款不在当前 MVP 范围

## 四、验证状态

- 自动化测试：`pytest -q` => `38 passed`
- 语法检查：`python3 -m compileall app tests` => 通过
- 手工验证：已验证入账幂等、冻结结算、cursor 稳定翻页、listener 重试与告警冷却

## 五、结论（管理视角）

当前项目完成度可以定义为：

- 核心业务链路：已完成
- 可外测准备：接近完成
- 生产级运维能力：部分完成

建议执行顺序：**先做 P0，再做 P1**。这会在最短时间内把 Relay 从“可运行原型”推进到“可控外测系统”。
