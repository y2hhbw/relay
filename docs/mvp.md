# Relay MVP 文档

实现进度请同步参考：`docs/mvp-implementation.md`

## 1. 产品定义

Relay 是一个给 AI agent 使用的 API 储值、支付与调用网关。

它解决的问题不是单纯的“链上支付”，而是让 agent 可以先完成一次稳定币储值，然后在没有信用卡、没有人工采购、没有一堆独立 API key 的情况下，持续购买并调用互联网服务。

一句话定义：

> 用稳定币一次储值，用统一网关持续消费 API，让 wallet 在产品层面充当 API 购买身份。

在 MVP 阶段，Relay 不是一个开放 marketplace，也不是一个 token 协议。它首先是一个集中式的 API Gateway，前面接收稳定币储值，后面代理调用多个 web2 API，并在系统内部完成余额管理、额度冻结和实际结算。

## 2. 第一批用户与核心场景

第一批用户不是所有“AI 用户”，而是两类最明确的人：

1. 构建 agent 的开发者
2. 已经有自动化 workflow 的 agent/机器人系统

他们的共同需求是：

- 希望按调用付费，而不是月付大套餐
- 不希望每接一个服务商就重新处理开户、支付和密钥管理
- 希望 agent 可以自动消费能力，而不是每次人工介入

MVP 最核心的使用场景可以压缩为一个：

> 一个 agent 先给 Relay 储值 USDC，之后持续通过统一接口调用搜索、OCR、LLM 等服务，Relay 自动检查余额、冻结额度、实际结算、转发请求并返回结果。

在这个阶段，API 服务商不是核心产品用户，而是供给侧。MVP 重点是先证明“agent 愿意用钱包余额购买 API 调用”这件事成立。

## 3. MVP 范围与不做什么

### MVP 必须包含

MVP 只做四件事：

1. 钱包储值
2. 预付余额账户
3. API Gateway 调用、额度冻结与结算
4. 三个首批 API 服务接入

对应的产品能力应该非常克制：

- 用户或 agent 获得一个储值地址
- 系统监听链上 USDC 储值并更新可用余额
- agent 通过统一接口查看可用服务与价格
- agent 发起调用时，系统先检查余额，必要时冻结一部分额度，再调用上游服务
- 上游返回真实 usage 后，系统按真实成本结算，并释放未使用额度
- 系统返回调用结果，并记录本次消费

### MVP 不做

下面这些内容都不进入第一个版本：

- 每次调用单独链上支付
- package credits
- 开放式 API marketplace
- agent identity network
- agent-to-agent service network
- 多链同时支持
- 复杂 token 设计

原因很简单：这些东西都会抬高设计成本和叙事复杂度，但不会显著提高你验证 MVP 的速度。

## 4. 最小可执行技术方案

### 4.1 技术决策

为了尽快做出可用版本，MVP 采用一条单一路径：

- 链：Base
- 稳定币：USDC
- 支付方式：只在储值时上链
- 账户模式：预付余额账户
- 结算方式：API 调用全部 off-chain，采用固定扣费或冻结后结算
- 后端：FastAPI
- 数据库：Postgres
- 限流与短期缓存：Redis
- 链上监听：`ethers.js` 或等价 EVM 监听方案

这里最关键的不是具体框架，而是结算模型：

> 储值上链，调用线下结算。

这是整个 MVP 能不能成立的核心。因为如果每次 API 调用都要求链上支付，成本、延迟和失败率都会高到不可用；而如果所有服务都要求调用前精确报价，LLM 这类按实际 usage 计费的服务又会很难处理。

### 4.2 系统结构

最小系统只需要五个组件：

1. Payment Listener
2. Balance DB
3. Billing Engine
4. API Gateway
5. Service Router
6. 上游 API Provider

逻辑关系如下：

```text
AI Agent
   |
   v
Relay API Gateway
   |-----------------> Balance DB
   |-----------------> Billing Engine
   |-----------------> Service Router
   |                         |
   |                         v
   |                    Search / OCR / LLM Provider
   |
   v
返回调用结果

链上储值 --> Payment Listener --> Balance DB
```

### 4.3 核心调用流程

储值流程：

```text
1. 系统为用户生成或分配储值地址
2. 用户向该地址发送 USDC
3. Payment Listener 监听到转账
4. 系统校验到账并增加可用余额
```

调用流程分成两类：

#### 固定价格服务

```text
1. Agent 调用 Relay 的服务接口
2. Gateway 读取该服务价格
3. Gateway 检查账户余额是否足够
4. 余额足够则先扣费
5. Gateway 调用对应上游 API
6. 返回结果并记录消费日志
```

#### 成本不确定服务（例如 LLM）

```text
1. Agent 发起调用，并带上可估算成本的参数，例如 model 和 max_tokens
2. Billing Engine 计算一个保守的最大成本上限
3. 系统检查余额是否覆盖这个上限
4. 如果足够，则先冻结该额度
5. Gateway 调用上游 LLM API
6. 上游返回真实 usage
7. 系统按真实成本结算
8. 未使用的冻结额度退回可用余额
9. 返回结果并记录消费日志
```

这样，用户只需要先储值一次，后续就可以持续调用；对于能提前定价的服务直接扣费，对于不能提前精确报价的服务先冻结、后结算。

### 4.4 首批 API

MVP 只接入三个 API：

1. LLM inference
2. Web search
3. OCR

这是一个比较合理的组合，因为它们覆盖了最常见的 agent 能力：思考、检索、读取外部信息。

### 4.5 最小安全要求

MVP 不需要做复杂安全体系，但至少要有：

- 短期签名 token 或 API key 认证
- 基于账户和服务的 rate limit
- 链上储值确认与幂等处理
- 调用日志、冻结日志与结算日志
- 上游 API 失败时的错误记录

第一个版本最重要的不是“绝对去中心化”，而是确保余额不会乱、冻结不会乱、结算不会乱、失败请求能追溯。

## 5. 这个产品的优点和缺点

### 优点

第一，这个方向比很多 AI + crypto 组合更真实，因为它对应的是一个已经存在的明确痛点：agent 有执行能力，但没有原生购买服务的能力。

第二，它的 MVP 可以做得非常小。你不需要先做协议、生态或市场网络，只需要做一个储值 + 结算 + 转发的系统，就能验证用户是否愿意为这种模式买单。

第三，它的商业模式简单。最容易落地的方式就是在上游 API 成本上加价，形成 markup，而不是一开始就设计复杂收费体系。

第四，它对用户的价值不只是“支持稳定币支付”，更是“统一接入和统一结算”。这一点更容易形成基础设施价值。

### 缺点

第一，这个 MVP 本质上是中心化系统。充值来自链上，但余额、扣费、路由和结算都在平台内部完成，所以信任模型并不去中心化。

第二，供给侧并不天然成立。即使需求侧愿意用，服务商也不一定愿意直接接入，所以前期很可能还是你自己作为聚合层去购买上游服务，再卖给用户。

第三，风控和运营比表面上重。你需要处理储值到账、冻结与结算争议、上游失败、价格波动、滥用调用等问题，这些都不是链能自动解决的。

第四，这个产品真正的护城河不会来自“能收稳定币”，而来自体验、成本控制、服务聚合和 SDK 集成。如果执行一般，很容易变成一个薄网关。

## 6. 为什么这个 MVP 路径最合适

这条路径的优点在于：它把最核心的验证问题提前了。

你现在真正需要验证的不是 marketplace 是否成立，也不是 agent economy 是否会爆发，而是下面三个问题：

1. agent 开发者是否愿意先储值再持续消费 API
2. 统一网关是否能让调用体验明显优于自己直连多个 API
3. 通过简单 markup 是否能覆盖基础成本并形成初步商业闭环

选择 `Base + USDC + 储值上链、调用线下结算`，就是为了最快验证这三个问题。

它牺牲了一部分“crypto 纯度”，但换来了更低复杂度、更快迭代速度和更高的工程可控性。对 MVP 来说，这是正确取舍。

## 7. 如何尽快实现 MVP

最短路径不是先做完整平台，而是按下面顺序做：

### 第一步：做最小余额系统

先完成三件事：

- 储值地址
- 链上到账监听
- 余额更新与查询

做到这一步，你已经验证了最基础的资金入口和预付余额模型。

### 第二步：只接一个 API 打通全链路

不要一开始接三个服务。先只接一个最容易验证价值的 API，建议从 `Web search` 或 `LLM inference` 二选一。如果先做 `LLM inference`，要同时把冻结后结算机制一起做进去。

目标不是功能丰富，而是证明下面这条链路成立：

```text
储值 -> 余额到账 -> 发起调用 -> 固定扣费或冻结后结算 -> 返回结果
```

### 第三步：补齐统一服务目录

当第一条调用链路跑通后，再补一个简单的服务目录接口，例如：

- 服务名
- 单价
- 描述
- 输入参数说明

这时候产品才开始像“统一网关”，而不只是一个单接口代理。

### 第四步：补两个额外 API

在第一个服务稳定后，再加入 OCR 和另一个基础能力。这样你既能验证多服务路由，也不会让系统一开始就太复杂。

### 第五步：做最小 SDK

SDK 不需要一开始就很重。第一个版本只需要帮助 agent 做三件事：

- 查询余额
- 查询服务
- 发起调用

如果首批用户主要是 Python agent，优先做 Python SDK；其他语言延后。

## 8. 建议的 MVP 验收标准

当下面这些条件同时成立时，可以认为 MVP 已经具备对外测试价值：

- 用户可以完成一次真实 USDC 储值
- 系统可以正确更新余额
- 用户可以通过统一接口调用至少一个上游服务
- 固定价格调用可以正确扣费并留存记录
- 成本不确定调用可以正确冻结额度、按真实成本结算并退回剩余额度
- 失败请求不会错误结算，或至少可以清楚追踪并人工处理
- 至少三个基础接口中的一个已经稳定可用

## 9. 第二阶段再考虑什么

如果 MVP 跑通，第二阶段再考虑下面这些扩展：

- 开放更多 API
- 对外提供服务商接入能力
- 增加 package credits
- 做更好的 agent SDK 和身份体系
- 探索 marketplace 或 reputation system

这些都应该建立在第一个问题已经被证明之后：

> 是否真的有人愿意用“先储值、后持续调用”的方式，让 agent 购买 API 能力。

## 10. 结论

Relay 最值得做的地方，不是它有 crypto 概念，而是它有一个足够具体、足够早期、也足够能验证的基础设施切口。

如果你现在就要开做，最正确的策略不是把想法继续扩大，而是把第一个版本压缩成：

> Base 上的 USDC 储值 + 平台内预付余额账户 + 固定扣费/冻结结算并存的统一 API 网关。

这样你会最快知道这到底是一个真实产品，还是一个只在叙事上成立的想法。
